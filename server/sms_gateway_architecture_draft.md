# 簡訊服務商（SMS Gateway）系統架構設計

## 1. 系統概述

本系統旨在建立一個高效能、可擴展的簡訊服務商（SMS Gateway），以支援大規模簡訊的批次發送。系統將充分利用 Go 語言在併發處理上的優勢，以及 Node.js 在 API 層快速開發的靈活性，實現兩者之間的協同工作。

## 2. 技術選型與分工

### 2.1 Go 語言的角色與優勢

Go 語言將作為系統的核心簡訊發送引擎，負責處理 SMPP 客戶端通訊、高併發簡訊處理以及批次發送邏輯。Go 語言以其卓越的併發處理能力（透過 Goroutines 和 Channels 實現）、高效能和低記憶體佔用而聞名，使其成為處理大量網路連線和簡訊發送任務的理想選擇。特別是在處理複雜的 SMPP 協議時，Go 能夠提供穩定且高效的表現，確保簡訊的快速可靠傳輸。

Go 語言的關鍵組件包括 **SMPP 客戶端**，它利用如 `gosmpp` 等成熟的函式庫與上游 SMSC 進行通訊，負責建立、維護和管理 SMPP 連線。**批次發送器**則從訊息佇列中高效地取出簡訊任務，進行批次處理，並透過 SMPP 連線將簡訊發送至電信網路。此外，**連線池管理**組件負責管理與多個 SMSC 之間的 SMPP 連線池，動態調整連線數量以應對不同的流量需求，確保高可用性和負載平衡。

### 2.2 Node.js 的角色與優勢

Node.js 將主要負責系統的 API 層、Web 管理介面、使用者認證與授權，以及簡訊提交與查詢介面。Node.js 憑藉其非阻塞 I/O 模型和豐富的生態系統，非常適合構建快速響應的 API 服務和直觀的使用者介面。它能夠輕鬆處理來自客戶端的大量請求，並將簡訊發送任務高效地提交給後端的 Go 引擎進行處理。

Node.js 的關鍵組件包括提供標準化接口供外部客戶端使用的 **RESTful API**，用於提交簡訊、查詢簡訊狀態、管理帳戶等。**Web 管理介面**則為營運人員提供一個友善的介面，用於監控系統狀態、管理使用者帳戶、查看詳細的簡訊發送報告等。最後，**認證與授權**組件負責實施 API Key 或 OAuth2 等認證機制，確保只有合法的請求才能存取系統資源。

## 4. 核心組件設計

### 4.1 訊息佇列 (Message Queue)

考量到簡訊發送可能面臨的流量峰值和對可靠性的嚴格要求，選擇一個高效能、持久化的訊息佇列至關重要。訊息佇列的主要作用在於解耦 Node.js API 層與 Go 發送引擎，提高系統的彈性；同時，它也能有效削峰填谷，緩衝簡訊提交的瞬時高峰，避免後端發送引擎過載。此外，訊息佇列還能確保簡訊在發送引擎暫時故障時不會丟失，並支援自動重試機制，從而提升整體系統的可靠性。

在選型方面，我們推薦以下兩種方案：

| 訊息佇列 | 優勢 | 適用場景 |
|---|---|---|
| **Kafka** | 高吞吐量、持久化、分散式、可擴展性強，適合處理大量即時數據流。 | 適用於需要處理極高併發簡訊提交、日誌收集、即時數據分析等場景。若系統規模預計會非常龐大，且對數據持久性和吞吐量有極高要求，Kafka 是理想選擇。 |
| **RabbitMQ** | 成熟穩定、功能豐富（如多種交換機類型、路由、QoS）、易於使用和管理。 | 適用於需要複雜路由、訊息確認、死信佇列等進階訊息處理功能的場景。對於中等規模的簡訊服務，或需要更精細控制訊息流的應用，RabbitMQ 是個不錯的選擇。 |

### 4.2 資料庫 (Database)

資料庫是簡訊服務商系統中不可或缺的組件，主要用於儲存簡訊發送記錄、使用者帳戶資訊、費率配置、簡訊模板以及系統日誌等關鍵數據。一個穩定、高效且可擴展的資料庫能確保數據的完整性、一致性和可查詢性，為系統的運作和營運分析提供堅實的基礎。

在選型方面，我們推薦以下兩種關聯式資料庫方案：

| 資料庫 | 優勢 | 適用場景 |
|---|---|---|
| **PostgreSQL** | 功能強大、高度可擴展、支援複雜查詢、資料完整性高、社群活躍。 | 適用於需要儲存大量結構化數據、支援複雜報表查詢、對資料一致性要求高的場景。特別適合儲存簡訊詳情、使用者帳戶、計費資訊等核心業務數據。 |
| **MySQL/TiDB** | MySQL 成熟穩定、廣泛應用、生態系統豐富；TiDB 則具備 MySQL 相容性，同時提供水平擴展和強一致性。 | MySQL 適用於大多數中小型簡訊服務，易於部署和管理。若未來預期數據量會爆炸性增長，且需要分散式資料庫的水平擴展能力，TiDB 作為 MySQL 生態的延伸，是更好的選擇。 |

### 4.3 API 設計 (Node.js)

Node.js API 層將提供一套清晰、標準化的 RESTful API 接口，供外部客戶端（如企業應用、網站後台）提交簡訊發送請求、查詢簡訊狀態、管理使用者帳戶等。API 設計應遵循 RESTful 原則，使用標準的 HTTP 方法（GET, POST, PUT, DELETE）和狀態碼，並採用 JSON 作為數據交換格式。為了確保安全性，所有 API 請求都應透過 HTTPS 進行，並實施嚴格的認證與授權機制。

**核心 API 接口範例：**

| API 端點 | HTTP 方法 | 功能描述 | 請求範例 (JSON) | 回應範例 (JSON) |
|---|---|---|---|---|
| `/sms/send` | `POST` | 提交單條或批次簡訊發送請求。 | `{"to": "+886912345678", "message": "您的驗證碼是123456", "sender_id": "MyCompany"}` | `{"status": "success", "message_id": "uuid-1234-abcd"}` |
| `/sms/status/{message_id}` | `GET` | 查詢指定簡訊的發送狀態。 | 無 | `{"message_id": "uuid-1234-abcd", "status": "DELIVERED", "delivery_time": "2023-10-27T10:30:00Z"}` |
| `/account/balance` | `GET` | 查詢當前帳戶餘額。 | 無 | `{"balance": 1000, "unit": "credits"}` |

### 4.4 監控與日誌系統

為了確保系統的穩定運行和快速故障排除，建立完善的監控與日誌系統至關重要。監控系統應能實時收集 Go 引擎的發送速率、佇列深度、SMPP 連線狀態，以及 Node.js API 的請求量、響應時間等關鍵指標。日誌系統則應記錄所有重要的操作事件，包括簡訊提交、發送狀態回報、錯誤訊息等，以便於追蹤和審計。

**推薦工具：**

| 類別 | 推薦工具 | 說明 |
|---|---|---|
| **監控** | Prometheus + Grafana | Prometheus 負責數據採集和儲存，Grafana 負責數據視覺化和儀表板展示。 |
| **日誌** | ELK Stack (Elasticsearch, Logstash, Kibana) | Elasticsearch 用於日誌儲存和檢索，Logstash 用於日誌採集和處理，Kibana 用於日誌視覺化和分析。 |
| **告警** | Alertmanager | 與 Prometheus 整合，根據預設規則發送告警通知（如郵件、Slack）。 |

## 5. 擴展性與可靠性考量

### 5.1 水平擴展

系統的各個組件都應設計為可水平擴展。Node.js API Gateway 可以透過負載均衡器（如 Nginx, Haproxy）在多個實例之間分發請求。Go 簡訊發送引擎也可以部署多個實例，共同消費訊息佇列中的簡訊任務，並管理各自的 SMPP 連線池。資料庫可以考慮採用主從複製或分散式資料庫解決方案（如 TiDB）來提升讀寫能力和可用性。

### 5.2 容錯與重試機制

*   **訊息佇列**：利用訊息佇列的持久化特性和確認機制，確保簡訊在處理過程中不會丟失。對於處理失敗的訊息，應將其重新放入佇列或轉移到死信佇列 (Dead Letter Queue) 進行後續處理。
*   **Go 發送引擎**：實作簡訊發送的重試邏輯，針對暫時性錯誤（如網路瞬斷、SMSC 暫時不可用）進行自動重試。設定合理的重試次數和間隔，避免無限重試造成資源耗盡。
*   **資料庫**：採用資料庫事務 (Transactions) 確保數據操作的原子性。定期備份資料庫，並建立異地備援機制。

### 5.3 流量控制與限速

為了保護上游 SMSC 不被過載，並確保系統的穩定性，Go 發送引擎應實作流量控制和限速機制。這可以透過令牌桶 (Token Bucket) 或漏桶 (Leaky Bucket) 演算法來實現，限制每秒發送的簡訊數量，確保發送速率符合 SMSC 的要求。

## 6. 安全性考量

*   **API 安全**：所有對外 API 接口必須透過 HTTPS 傳輸，防止數據竊聽。使用 API Key 或 OAuth2 進行認證和授權，並定期輪換 API Key。
*   **數據加密**：敏感數據（如使用者密碼、API Key）在資料庫中應加密儲存。簡訊內容在傳輸過程中也應考慮加密。
*   **輸入驗證**：對所有來自客戶端的輸入進行嚴格的驗證和過濾，防止 SQL 注入、XSS 等常見安全漏洞。
*   **權限管理**：實施最小權限原則，確保系統各組件和使用者只能存取其完成工作所需的最小資源。

## 7. 部署與維運

*   **容器化**：建議使用 Docker 將 Node.js API Gateway 和 Go 簡訊發送引擎進行容器化，便於部署、管理和擴展。
*   **容器編排**：使用 Kubernetes 等容器編排工具來自動化部署、擴展和管理容器化應用。
*   **CI/CD**：建立持續整合/持續部署 (CI/CD) 流程，自動化程式碼測試、建置和部署，提高開發效率和部署頻率。
*   **監控與告警**：如前所述，建立完善的監控與告警系統，實時掌握系統運行狀態，及時發現並解決問題。

## 8. 總結

本架構設計提供了一個基於 Go 語言和 Node.js 的高效能、可擴展的簡訊服務商系統藍圖。透過合理的技術選型、組件分工和安全性考量，旨在構建一個穩定可靠、易於維護和擴展的簡訊發送平台。
